# ARP Analysis Practical

## 🎯 Objective

Capture and analyze ARP traffic using Wireshark to understand how devices resolve IP addresses into MAC addresses before communicating on a local network.

---

# Prerequisites

Before starting this practical, ensure that:

- Wireshark is installed.
- Npcap is installed.
- Your system is connected to a Local Area Network (LAN).
- You have Administrator privileges.

---

# Step 1: Open Wireshark

Launch Wireshark.

Select your active network interface.

Examples:

- Wi-Fi
- Ethernet

---

# Step 2: Start Packet Capture

Double-click the active network interface to begin capturing packets.

---

# Step 3: Generate ARP Traffic

Open **Command Prompt**.

First, view the current ARP cache:

```bash
arp -a
```

Then generate ARP traffic by pinging your default gateway.

Example:

```bash
ping 192.168.1.1
```

> Replace the IP address with your router's IP address if it is different.

---

# Step 4: Stop Packet Capture

After the ping command completes, click the **Red Stop** button to stop packet capture.

---

# Step 5: Apply Display Filter

In the Display Filter bar, type:

```text
arp
```

Press **Enter**.

Only ARP packets should now be displayed.

---

# Step 6: Analyze the ARP Request

Select an ARP Request packet.

Expand the **Address Resolution Protocol** section.

Observe:

- Sender MAC Address
- Sender IP Address
- Target MAC Address
- Target IP Address
- Opcode (Request)

---

# Step 7: Analyze the ARP Reply

Select the corresponding ARP Reply packet.

Observe:

- Sender MAC Address
- Sender IP Address
- Target MAC Address
- Target IP Address
- Opcode (Reply)

Verify that the requested MAC address has been returned.

---

# Step 8: Verify the ARP Cache

Return to Command Prompt.

Run:

```bash
arp -a
```

Verify that the IP-to-MAC mapping now appears in the ARP cache.

---

# Expected Result

You should successfully identify:

- ARP Request
- ARP Reply
- Sender IP Address
- Target IP Address
- Sender MAC Address
- Target MAC Address

# Observation

- The sender first checked its ARP cache.
- Since the MAC address was not available (or needed refreshing), an ARP Request was broadcast.
- The destination device replied with its MAC address.
- The operating system stored the mapping in the ARP cache for future communication.

---

# Conclusion

In this practical, ARP traffic was successfully captured and analyzed using Wireshark. The ARP Request and ARP Reply packets demonstrated how devices resolve IP addresses into MAC addresses before communication on a local network. The ARP cache was also verified to confirm that the mapping had been stored successfully.
